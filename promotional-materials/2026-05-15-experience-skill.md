# experience-skill：为 AI Agent 构建可生长的长期记忆

> 告别每次查询从零开始的检索模式。experience-skill 将 LLM Wiki 理念工程化落地，用数据库索引替代文件索引，让 AI 助手真正具备积累和复用知识的能力。

---

## 背景

2026 年 4 月，Andrej Karpathy 在 [LLM Wiki](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f) 中提出了一个重要的理念转变：与其让 LLM 在每次查询时从原始文档中重新检索和拼凑答案（传统 RAG 模式），不如让 LLM 维护一个持久化的 Wiki——知识被编译一次，持续更新，反复复用。

这个理念迅速获得了社区的广泛认同，gist 两周内收获 5000+ star，Kompl、Link、Synthadoc、OmegaWiki 等一批开源实现相继涌现。然而，在社区实践中，一个关键的工程瓶颈逐渐浮出水面：Karpathy 原型方案中的文件级索引（index.md）在 Wiki 规模从几十页增长到几百页时，检索效率和可维护性都面临挑战。

experience-skill 正是为解决这一瓶颈而设计。

---

## 核心思路：用数据库索引替代文件索引

Karpathy 原型方案中，Wiki 的检索入口是一个 `index.md` 目录文件。LLM 每次回答问题，需要先将整个 index 读入上下文，从中找到相关页面链接，再逐个钻入对应的 .md 文件。这在文档数量较少时运转良好，但随着经验库的积累，三个问题逐渐凸显：

- **索引膨胀**：index.md 随文档数线性增长，LLM 需一次性加载完整目录才能定位目标
- **维护成本**：每新增一篇文档，index.md 都需由 LLM 参与更新，消耗 token 且引入出错风险
- **检索能力受限**：无法按类型、关键词等维度做结构化筛选，只能依赖 LLM 对目录全文的语义理解

experience-skill 的方案是将 `index.md` 替换为 **SQLite + FTS5 全文搜索引擎**。这是一个看似简单、实则关键的工程决策：

- **毫秒级检索**：FTS5 在数据库层面完成全文匹配，返回最相关的 N 条结果，LLM 仅需读取命中条目，无需加载整个知识库
- **混合检索策略**：默认模式下，FTS5 元数据搜索（name、description、keywords）与正文 grep 加权融合，兼顾精确匹配与语义覆盖
- **结构化查询**：支持按类型（Skill/Wiki）、关键词、热门程度等多维筛选，这些对 SQL 是基本功
- **内置中文分词**：通过 C 语言实现的 simple tokenizer 扩展，中文检索精度不依赖 LLM 理解

对于终端用户而言，这些技术细节完全透明。用户像平常一样用自然语言与 AI 助手交流，Agent 在后台自动检索经验库，命中后将过往经验作为回答的主要参考依据。整个过程无需学习新工具或记忆任何命令。

## 产品形态：一个 Agent Skill，而非独立服务

experience-skill 在架构上选择了 Agent Skill 的产品形态，而非 MCP Server 或独立服务。这意味着零集成成本：将其放入 Agent 的 skills 目录后，Agent 框架自动发现并加载其行为指令，按 SKILL.md 中定义的流程执行。

这种设计带来了几个实际好处：

- **自动发现与加载**：无需配置 API 端点、无需维护后台进程
- **与其他 Skill 共存**：Agent 根据任务内容自动路由到相应 Skill，互不干扰
- **原生工具调用**：Agent 通过 `search-experiences` 检索知识库，与 `grep`、`ls` 等 Shell 工具平等对待，无需特殊适配

experience-skill 的核心价值在 Agent 会话中释放——用户无论通过终端、桌面应用还是 Web 界面与 Agent 交互，Agent 都会自动检索本地经验库，基于过往积累的知识回答问题。整个过程对用户完全透明：你只需要像平时一样用自然语言提问。

以下是 experience-skill 在 OpenCode 中的真实工作过程。

**知识沉淀**：用户提供一批原始运维案例，Agent 自动逐篇分析、归类、去重，批量生成结构化的 Skill 与 Wiki 并导入数据库。用户只需给出方向性反馈，Agent 即迭代调整分类框架、关键词和评测用例，全程无需手动编辑文件。

![知识沉淀](assets/2026-05-15-experience-skill/agent/01-知识沉淀.png)

**质量优化**：Agent 可对全部经验执行五维结构化评估——准确性、完整性、可读性、时效性、可执行性——逐一打分后自动补全缺失项，包括可执行脚本、兼容性元数据、关键词覆盖和评测用例，无需人工逐项检查。

![质量优化](assets/2026-05-15-experience-skill/agent/02-质量优化.png)

**并行审计**：面对大规模经验库的全面检查任务，Agent 自动拆解为多个并行子任务，同时读取各目录结构、配置文件和评测用例，最终汇总为一份结构化报告，无需人工逐个检查。

![并行审计](assets/2026-05-15-experience-skill/agent/03-并行审计.png)

## 极致轻量

experience-skill 的技术栈仅包含 Python、SQLite 以及一个可选的 C 语言中文分词扩展。不需要 Elasticsearch，不需要向量数据库，不需要 Docker，不需要任何云端服务。

104 条经验（含全文索引）的数据库文件约 480KB——一张手机截图的体积都远大于此。Linux、macOS、WSL 均可运行，`uv sync` 完成依赖安装，`uv run experience-skill web` 启动管理界面。

单文件数据库意味着知识库可以像普通文件一样管理：U 盘带走、Git 版本控制、rsync 迁移——它就是一个文件。

## 检索质量：百篇规模实测

为了验证 FTS5 方案在真实场景下的检索效果，我们在 104 条经验（21 条 Skill + 83 条 Wiki，数据库 480KB）上运行了完整的基准测试，并引入文件级 grep 方案作为对照组。

**Skill 检索（21 条 Skill × 22 个查询）：**

| 检索方式         | 查询数 | Top-1 命中率 | Top-3 命中率 | Top-5 命中率 | 平均耗时 |
| ---------------- | ------ | ------------ | ------------ | ------------ | -------- |
| FTS5 元数据检索  | 22     | 100.0%       | 100.0%       | 100.0%       | 5.0 ms   |
| 正文 grep        | 22     | 100.0%       | 100.0%       | 100.0%       | 45.9 ms  |
| 混合检索（默认） | 22     | 100.0%       | 100.0%       | 100.0%       | 51.4 ms  |
| 文件 grep 基线   | 22     | 100.0%       | 100.0%       | 100.0%       | 0.8 ms   |

Skill 数量较少（21 条），各方案均能在 Top-1 命中。FTS5 仅需 5ms。区分度体现在 Wiki 场景。

**Wiki 检索（83 条 Wiki × 10 个查询）：**

| 检索方式         | 查询数 | Top-1 命中率 | Top-3 命中率 | Top-5 命中率 | 平均耗时 |
| ---------------- | ------ | ------------ | ------------ | ------------ | -------- |
| FTS5 元数据检索  | 10     | 80.0%        | 90.0%        | 100.0%       | 3.9 ms   |
| 正文 grep        | 10     | 70.0%        | 100.0%       | 100.0%       | 44.9 ms  |
| 混合检索（默认） | 10     | 80.0%        | 100.0%       | 100.0%       | 52.1 ms  |
| 文件 grep 基线   | 10     | 60.0%        | 100.0%       | 100.0%       | 2.4 ms   |

在 83 篇文档中定位目标，混合检索 Top-1 命中率 80%，MRR 88.3%，分别比文件 grep 基线（60%、78.3%）高出 20 和 10 个百分点。这意味着文件级 grep 方案有 40% 的查询无法在首位返回最匹配的文档——排在后面的噪音条目会消耗 LLM 大量上下文窗口。

## 定位

experience-skill 并非又一个 RAG 管道，也并非又一个通用 Wiki 工具。它在三个方向之间找到了自己的位置：

|                | 传统 RAG         | Karpathy LLM Wiki     | experience-skill      |
| -------------- | ---------------- | --------------------- | --------------------- |
| 知识积累方式   | 每次查询重新检索 | 持久化 Wiki，增量更新 | 持久化 Wiki + DB 索引 |
| 检索机制       | 向量相似度       | LLM 读 index.md       | SQLite FTS5           |
| 部署要求       | 向量数据库       | 无                    | 无                    |
| Agent 接入方式 | API / MCP Server | 手动配置              | 自动发现（Skill）     |
| 存储成本       | 高（向量嵌入）   | 极低                  | 极低                  |

继承 LLM Wiki 的持久化和可累积性，借鉴传统 RAG 的快速检索能力，最终封装为一个 Agent Skill——让它成为 AI 助手能力的一部分，而非需要单独维护的外部系统。

---

## 快速开始

将 experience-skill 放入 Agent 的 skills 目录，安装依赖后，Agent 会自动加载。用户以自然语言提问，Agent 在后台检索经验库，匹配过往经验，基于本地知识回答问题。

如需人工管理经验库，启动 Web 界面即可浏览全部内容。CLI 工具链提供 `add-experiences`、`search-experiences`、`sync`、`list-experiences` 等完整的管理能力。
